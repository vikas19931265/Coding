Version 7.0.2-5 (based on ContactManager 7.0.2 with Diamond-17)

Enhancements and Fixes

  • General

    • The Contact tab now has a dropdown menu that lists the last X contacts the user has visited (from most recent to least recent). X is the value of the MaximumViewedContacts parameter, which initially is set to 10. This makes it easier for students to jump between contacts they have recently viewed without having to re-search for the contacts.

    • An "AllowActiveX" parameter has been added to config.xml to suppress an error message some students get that states "Failed to load TemplateRunner ActiveX 32 bit control. Document editing will fail.".

    • The DuplicateContacts batch process is scheduled to run twice a day by default. This results in an error during startup about "no or unmodifiable schedule found" for this process. To prevent the error, the batch process has been unscheduled.

    • Typos were fixed in comments for:

      • ta.QueryUtil findContactByID() method

      • trainingapp.base findContactByID() method

    • Entries added to ExampleList:

      • UI Containers -> Comment (in PCF element)

      • Cross-Section Functionality -> Entity name, datetime used in

  • Data Model

    • In the DoctorSpecialtyType typelist, for the Unspecified typecode, the various Categories entries have been replaced by a single Category List entry.

  • User Interface

    • In ContactBasicDetailsDV.ABPerson, the First Name field is marked as required. In TrainingApp, this container is used only when creating new person contacts. This change was made to avoid the minimum contact creation criteria error message that talks about contact tags.

    • The "Delete" button that lets you delete contacts from search results on the Search tab is now working. (The ExposeIterator tag in ContactSearchResultsLV has been restored.)

    • A comment was added to VendorEvaluationDV's TotalScore widget identifying that the logic used on the reflection tab is duplicated in the VendorEvaluationEnhancement TotalScore property.

  • Gosu

    • The code that creates "User viewed this contact" history events has been corrected so that the history events get committed to the database correctly.

    • The logic in the following business rules has been changed to use hasMatch (the "preferred does-this-exist condition checker") instead of exists:

      • ABPU1030 - Flag invalid address

      • ABPU2010 - Flag lack of email address

    • In VendorEvaluationEnhancement's TotalScore property, the comment was modified to identify that the logic for TotalScore is also coded on the VendorEvaluationDV's TotalScore field's reflection tab, and any change made to the virtual property should also be made to that widget's reflection logic.

    • Some of the comments in the business rules were cleaned up.

  • Cross-Section Functionality

    • On ABContactDetailsCompanyDV, if a company's primary contact has no email, the "Prefers Contact By Email" is never available. (Previously, it was available if you clicked Update for a company whose Preferred Contact had no email address.)

    • On ABContactDetailsCompanyDV, the "Relation to Finance Manager" reflection has been modified to be bi-directional. (Reflection works if either field is changed.)

    • The entity name for ContactNote now checks to see if Body is null, and if it is it uses the string "(no body text)".

Version 7.0.0.501-3 (based on ContactManager 7.0.0 build 501 with Diamond-9)

Enhancements and Fixes

  • General

    • On the ABContactSummaryDV, the "ID" field (which mapped to ABContact.ID) has been replaced with a "Public ID" field (which maps to ABContact.PublicID).

    • Several additional findContact() methods have been added to ta.QueryUtil to enable searches for all base application ABContact subtypes without needing to subtype the results. For example, a few of the new methods are: findPerson(), findDoctor(), findAttorney(), and findCompany(). Each of these methods takes a public ID and returns the ABContact with that ID.

    • The Admin tab has a new menu item: "Load TrainingApp Sample Data". This menu item loads the ContactManager sample data and then deletes the orphan messages. Previously, this functionality was split across two menu items, "Load Sample Data" and "Delete Orphan Messages". These old menu items have been removed.

    • The Contact Assignment -> Assignment Transfer screen has been enhanced. Now, when you choose the current and target contact, you get a list of contacts assigned to that user.

    • The ABPerson enhncement's Age getter was improved. Previously, it determined age by dividing number of days since birth by 365. This produced inccurate results because of leap year days. The rule now correctly calculates the difference between the birth year and the current year, and subtracts 1 if the person's birthday has not yet occurred this year.

    • ABContactPreupdate rule ABPU2020 (Depart from company as primary contact) has been removed. Previously, this was an example of a "disabled rule". But the rule did not work once enabled, and instructor feedback suggested that the example wasn't really needed, as one can easily demo activating and dectiving rules. So the rule was simply removed.

  • Integration

    • Virtually all code that manipulated bundles via getCurrent()/commit() has been modified to instead make use of runWithNewBundle(). One example of getCurrent()/commit() was left (the one for assigned user reassignment). The modified functionality includes:

      1. Recording each contact visited

      2. Toggling a vendor status from recommended to not recommended

      3. Message acknowledgement page

      4. Code to report duplicate messages

      5. Message Exception page

      6. Message Generator page

      7. Legal Case Report messaging (ABContactDetailsPersonDV)

    • The logic for toggling a vendor status was moved from the ABContactEnhancement to a PCF function on the ABContctDetailsCV file. This is because the logic must commit changes to the ABContact, and it's generally not good practice to have an enhancement function commit changes to the target instance, as one can't be sure about the context from which an enhancement method is called. In this case, it's better practice to put this logic with the thing that calls it.

    • The examples in the trainingapp.demo.gosu.BundleExamples have been modified to set the ABCompany inspection date (as opposed to incrementing the ABCompany EmployeeSatisfactionScore). This was done because (1) the InspectionDate field is exposed in the UI, whereas the EmployeeSatisfactionScore field is not, (2) the code needed to set inspection date is shorter and more readable, and (3) the use case for inspection date is slightly more intuitive.

    • The "Training: Gosu" location group and all of its child elements have been removed. This location group was intended to showcase Gosu functionality in the UI, but in practice it makes more sense to run the Gosu examples from the Gosu Tester. Furthermore, the Gosu Bundles lab was modified to not make use of the Bundles Lab page. To ensure future builds are easier to upgrade, this vestigal functionality was removed.

    • There has always been a Java version of AcmeIExchangeRateSetPlugin in the TrainingApp base application. But, in previous builds, the logic in the Java class did not match the logic in the Gosu class. (The Java logic reflected an earlier draft of the plugin logic.) This has been corrected, and the Java class is now identical in logic to the Gosu class.

    • The following modifications were made to the VendorRecommendation messaging integration point:

      • The Gosu template now has a "SenderRefID: <@@senderRefID@@>" line.

      • The request plugin now sets the value of SenderRefID in the template.

      • The transport plugin now sends the transformed payload. (In earlier builds, it erroneously sent Message.Payload, which would ignore any changes made to the payload in the request plugin.)

    • The following modifications were made to the Legal Report messaging integration point:

      • The request plugin now adds the SenderRefID to the payload.

      • The ExternalApp web service has been changed to take only one parameter (the transformed payload) as opposed to two (the SenderRefID and the transformed payload). Conseqeuntly, the call to the web service passes only the payload.

      • The transport plugin now sends the transformed payload. (In earlier builds, it erroneously sent Message.Payload, which would ignore any changes made to the payload in the request plugin.)

    • All the examples in the "Overview of Advanced Gosu" lesson are now also in TrainingApp. This includes:

      • trainingapp.demo.gosu.Author (custom annotation)

      • trainingapp.demo.gosu.ColorType (enum)

      • trainingapp.demo.gosu.GenericsExample (class using generics)

      • trainingapp.demo.gosu.MiscellaneousExamples (intervals, null-safe operators, using custom annotations, lists, hashmaps, using generics, using typecode mapper)

      • Tests -> SampleTest -> BirthdayTest (GUnit test)

      • modules/configuration/config/examples/xml (XmlElement)

      • modules/configuration/config/typelists (Typecode Mapper)

Enhancements and Fixes

  • General

    • TrainingApp now sports the Diamond "look-and-feel". (ContactManager hasn't moved its login page to Diamond, so the TrainingApp login page is still Carbon-esque. This should get fixed in TrainingApp build 2.)

    • In the ContactManager base application, the ABContact entity has a Tags array that must contain at least one ABContactTag. In other words, every ABContact must have at least one tag associated to it. (A tag identifies the relationship between the contact and the customer and allows for contacts to have multiple relationships, such as being a lawyer vendor and a policy holder.) This functionality potentially compromises the intuitiveness of TrainingApp, so it has been hidden in the following ways:

      • The NewContact page references a set of modal detail views (such as ContactBasicsDV.ABCompany). All Tag fields in these modal views have been disabled.

      • The ABContact enhancement has a new method unique to TrainingApp: addDefaultTagToNewContact(). This adds a default tag (the "claim party" tag) to the newly created contact. It is called by the New Contact page's afterEnter() property, though it could be used any other time you want to create a contact through code.

      • The Tag fields on the various Search screens have been removed. (This search criteria is optional.)

    • TrainingApp now includes Windows shortcuts in c:/Guidewire/TrainingApp to start TrainingApp, stop TrainingApp, and start Studio.

    • TrainingApp now includes a dbbackup directory that contains a copy of the databases in their just-installed state. If you need to reset the database, you can drop the database and then copy over these files (which may be faster than dropping the database and then reloading the sample data).

    • TrainingApp now has a younger brother - ExternalApp. ExternalApp is a modified version of ContactManager that simulates multiple external systems. It is used primarily for the integration courses. TrainingApp now includes some integration points to ExternalApp. For more information on ExternalApp, refer to the ExternalApp build notes and example list.

In other words, a change to any bank account causes Preupdate rules to be run for the ABContact. You can see this in action in ABContact Preupdate rule ABPU1040. When a bank account is changed, its IsVerified flag should be set to false. But, there are no preupdate rules for bank accounts. So the unverify logic is put in the ABContact preupdate rules, and the triggerValidation tag is used to ensure that changes to bank accounts trigger the ABContact preupdate rules.

Enhancements and Fixes

  • General

    • Shortcuts have been added to start and stop TrainingApp, and to start Studio.

  • Data Model

    • All fields of type money in the FinancialSummary and Organization entities have been modified to be of type currencyamount; currencyamount is the recommended datatype for monetary fields for the following reasons:

      • The money datatype does not honor currency formatting rules; for example, if a field of type money displays 100 Japanese Yen, it will display 100.00 rather than the correct 100 with no decimal places.

      • The money datatype is not compatible with multicurrency environments; if an environment ever changed from a single-currency model to a multi-currency model all fields of type money would have to be modified.

    • ABContactSummaryDV has been modified to use the label property for the list view input that references FlagEntriesLV instead of using a label input.

      • The logic of the RecentlyViewedContactUtil class has been simplified.

Regenerating the Toolkit and Sample Data (v5.0 builds)

The dev-import command uses information in toolkit\bin\server.properties. In particular, it uses the port number specified in this file. Initially, the port number is set to 8280. Because TrainingApp runs on 8880, this port number must be manually reset. In an initial installation, one does not need to worry about this file. But, if for some reason one regenerates the toolkit and then attempts to import the sample data, the import will fail because it will be using the wrong port number.

Therefore, if for some reason you regenerate the toolkit, you must also reset the port number in server.properties to 8880.
