Multi-threaded Inbound Integration Overview

There are sometimes situations that require high-performance data throughput for inbound integrations that require special threading or transaction features from the hosting J2EE/JEE application environment. It is difficult to interact with the application server’s transactional facilities and write correct, thread-safe, high-performing code. ClaimCenter includes tools that help you write such inbound integrations. You can focus on your own business logic rather than how to write thread-safe code that works safely in each application server.
Inbound Integration Plugin
The highest-level part of the inbound integration system is a plugin interface called InboundIntegrationPlugin. The InboundIntegrationPlugin plugin interface defines the contract between ClaimCenter and inbound integration high-performance multi-threaded processing of input data.
To write your own implementation of this plugin interface, see “Custom Inbound Integrations”.
Depending on what you want to do, you might not need to write your own implementation of the plugin. To support common use cases, ClaimCenter includes two useful plugin implementations that are supported for production servers:
•
File inbound integration – A built-in implementation of InboundIntegrationPlugin that reads text data in local files. Poll a directory in the local file system for new files at a specified interval. Send new files to integration code and process incoming files line by line, or file by file. See “Inbound File Integration”. You provide your own code that processes one chunk of work, either one line, or one file, depending on how you configure it.
•
JMS inbound integration – A built-in implementation of the plugin that reads inbound messages from a JMS message queue. See “Inbound JMS Integration”. You provide your own code that processes the next message on the JMS message queue.
You must register the built-in plugin implementation in the Studio plugin registry. See “Registering a Plugin Implementation Class”. When registering your plugin implementation, you must also add one plugin parameter called integrationservice. That integrationservice parameter defines how to find configuration information in the inbound integration configuration XML file for that integration. See “Inbound Integration Configuration XML File”.
Inbound Integration Handlers (Only for File and JMS Inbound Integrations)
No matter whether you use the built-in integrations or write your own, you must define your code that handles one chunk of data.
If you use the built-in file or JMS inbound integrations, you must write a class called a handler for processing one chunk of data. ClaimCenter defines a plugin interface called InboundIntegrationHandlerPlugin. The InboundIntegrationHandlerPlugin interface contains only one method, which is called process. That method must handle one chunk of data, passed to the method as an argument of type java.lang.Object.
Write a handler class that implements the InboundIntegrationHandlerPlugin interface and implement the process method. Downcast the Object to the necessary type such as a Java file reference or a JMS message before processing the object. Next, register the plugin in the Studio plugin registry. See “Registering a Plugin Implementation Class”.
IMPORTANT When registering your plugin implementation, you must also add one plugin parameter called integrationservice. That plugin parameter links your plugin implementation to one XML element within the inbound-integration-config.xml file. See “Inbound Integration Configuration XML File”.
If you write your own implementation of the InboundIntegrationHandlerPlugin, you do not need to use or implement InboundIntegrationHandlerPlugin. Instead, you implement different types of code to identify the units of work and process them with code called work agents. See “Custom Inbound Integrations”. If you write your own plugin implementation for this plugin interface, you must use Java, not Gosu.
Inbound Integration Configuration XML File
In Studio, there is a configuration file for inbound integrations called inbound-integration-config.xml. In this file, define configuration settings for every inbound integration that you want to use. Each inbound integration requires configuration parameters such as references to the registered plugin implementations. See “Inbound Integration Configuration XML File”.
