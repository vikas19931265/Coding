Changes to Existing Web Services

For details of converted WS-I web services, refer to the documentation topic for each web service as specified in the following table. Note that all RPCE web services are deprecated in version 8.0. Begin to convert any RPCE web service code to WS-I.
For any web services that converted from RPCE to WS-I, the WS-I versions use different types so there will be code changes. For example, any arguments or return values that took entity types now are WS-I variants based on Gosu classes with those fields or XSD types.

ContactAPI
n/a
 
Manipulates contact data. For customers who use ContactManager, ContactManager calls this web service.
•
In version 7.0, some methods took a transaction ID, which represents a unique identifier for this request. If the request is duplicated later, the application knows that it is a duplicate and does not repeat the request. In 8.0, the methods no longer have the transaction ID as an explicit method argument. Instead, the methods have the annotation @WsiCheckDuplicateExternalTransaction, which automatically enforces unique transaction IDs based on SOAP headers. This change affects the methods updateContact, removeContact, and mergeContacts. See “Checking for Duplicate External Transaction IDs” in the Integration Guide. If the web service client is another Guidewire application, you can set the transaction ID with the API ContactAPIUtil.setTransactionId().
•
New methods required by the ABClientAPI, which the web service implements: pendingUpdateRejected, pendingCreateRejected, pendingCreateApproved, pendingUpdateApproved. These methods manage pending updates but the implementation is different for each Guidewire application.
In ClaimCenter, those methods create new activities for pendingUpdateRejected and pendingCreateRejected. For pendingCreateApproved, the application simply logs the request. For pendingUpdateApproved the method updates the ClaimCenter contact graph entities that the update operation created. The AddressBookUID values need to be updated with the values that ContactManager assigns.
See “Contact Web Service APIs” in the Integration Guide.
ImportToolsAPI
IImportToolsAPI
IUserAPI
IGroupAPI
In addition to converting IImportToolsAPI from RPCE to WS-I, the ImportToolsAPI web service has new methods:
•
importXml – For administrative data only, this method imports data from XML conforming to either an archiving import xsd, or from cc_import.xsd.
•
importArchiveXmlData – new method to import administrative data from XML data in UTF-8 encoding of archiving data format XML.
•
importArchiveXmlDataAsByteArray – new method to import administrative data from XML data in a byte array of archiving data format XML
From WS-I web services if you want to add a new user or group, use the WS-I web service ImportToolsAPI. Do not use the old web services specific to users and groups: IUserAPI and IGroupAPI. You can add a user or group with methods that import XML or CSV formatted text.
For other methods in the RPCE web services IUserAPI and IGroupAPI, there is no equivalent WS-I web service. If you used IUserAPI or IGroupAPI, Guidewire recommends that you write your own custom WS-I web service that transfers only the necessary data for each integration point.
LoginAPI
ILoginAPI
(however, see note)
Although the names are similar, the RPCE web service ILoginAPI and the WS-I web service LoginAPI are actually notably different.
The RPCE web service ILoginAPI had a critical role in authenticating with the server for later API calls with other web services. If you use the generated Java libraries and the included APILocator utility class, APILocator indirectly uses ILoginAPI.
The WS-I web service LoginAPI does not have the analogous role in WS-I authentication. In effect, WS-I authentication happens with each API call. However, if your web service client code wants to explicitly test specific authentication credentials, ClaimCenter publishes the built-in LoginAPI web service. Call this web service’s login method, which takes a user name as a String and a password as a String. If authentication fails, the API throws an exception. You can also use LoginAPI to purposely leave a user session open for logging purposes.
See “Login WS-I Authentication Confirmation” in the Integration Guide.
MaintenanceToolsAPI
IMaintenanceToolsAPI
Tools related to managing the application. There are various changes in 8.0 in addition to conversion to WS-I:
•
markForPurge – old deprecated method removed, replaced by markPurgeReady.
•
markPurgeReady – New additional parameter. If the value is true, then the application purges each claim even if it is part of an aggregate limit. If false, application does not purge claims in an aggregate limit, and for any such claims the method returns an error.
•
New methods for scheduling aggregate limits: scheduleAggLimitRebuildOfClaims, scheduleAggLimitRebuildOfPolicies, scheduleAggLimitRebuildAllLimits, scheduleAggLimitRebuildInvalidLimits.
•
scheduleForArchive – schedules for archive
•
restore – restores from archive
•
terminateBatchProcessByName changed name to terminateBatchProcessByName
•
terminateBatchProcessByID changed name to requestTerminationOfBatchProcessByID
•
stopWorkQueueWorkers – new method to stop query workers
•
The old getQueueWorkerStatus method (which was already deprecated) and getWorkQueueStatus were removed. The getWQueueStatus method replaces them and applies to whole cluster like the old getWorkQueueStatus method.
getNumActiveWorkItems – new method to get number of active work itemsSee “Maintenance Web Services” in the Integration Guide.
MessagingToolsAPI
IMessagingToolsAPI
Tools related to messaging, such as suspending a destination. There are various changes in 8.0 in addition to conversion to WS-I:
•
configureDestination – new method to configure some performance and error handling aspects of a messaging destination on a running server. Also see the getConfiguration method.
•
getConfiguration – get performance and error handling parameters from a messaging destination on a running server. Also see the configureDestination method.
There is also a new messaging_tools command line tool with similar functionality. Use the -config parameter to get the configuration for a destination. Use the -restart parameter to restart the messaging destination with new settings. If you use -restart, add additional options: wait, retries, initial, backoff, poll, threads, chunk.
See “Messaging Tools Web Service” in the Integration Guide.

SystemToolsAPI
ISystemToolsAPI
There are various changes in 8.0 in addition to conversion to WS-I:
•
getDBCCState – new method that returns the state of the consistency checks process.
•
submitDBCCBatchJob – new method that submits the consistency checks batch job on the underlying physical database
•
submitUpdateStats – new method that runs the update stats process on the underlying physical database. Afterwards, use the isUpdateStatsRunning method to check on the process status.
•
cancelUpdateStats – new method that cancels the update stats process if running. Afterwards, use the getUpdateStatsState method to check on the process status.
•
getUpdateStatsState – new method that checks whether the update stats process is currently running.
•
submitDmvReportBatchJobWithParams - the previous version used to take three parameters, and now takes only the includeDatabaseStatistics argument.
•
getRecentAWRSnapshotInfo - method now returns DatabaseSnapshotInfo[] instead of String
•
updateLoggingLevel -- logging level was passed as String, now is LoggingLevel
•
submitAwrReportBatchJob and submitAwrReportBatchJobAllOpts – new method to submit Oracle AWR Performance Report batch jobs.
•
getPerfReportsInfo – Retrieve the information about the latest Oracle AWR downloads.
See “System Tools Web Services” in the Integration Guide
