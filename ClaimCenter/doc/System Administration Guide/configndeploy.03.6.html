ClaimCenter System Administration Guide : Basic Configuration : Using the Geocoding Feature

Using the Geocoding Feature

Guidewire supports geocoding within ClaimCenter and ContactManager. The geocoding process assigns latitudes and longitudes to addresses. Software then uses geocoded addresses to present users with geographic information, such as the distance between two addresses. All primary addresses in ClaimCenter and ContactManager are candidates for geocoding.

Working with the Geocode Plugin

To implement geocoding, ClaimCenter provides the GeocodePlugin. Implementations of the plugin connect with specific external geocoding services, which provide geocode coordinates for specific addresses. Typically, plugin implementations also use an external mapping service to calculate and return proximity information, driving instructions, and maps. Guidewire Studio lets you enable the GeocodePlugin, specify which implementation to use, and specify parameters for the implementation you choose.
ClaimCenter provides a fully functioning and supported GeocodePlugin implementation, the BingMapsPlugin Gosu class. This plugin implementation connects to the Microsoft Bing Maps Geocode Service. If you intend to use the BingMapsPlugin implementation, your organization must have a valid account with Microsoft.
For more information about implementing the GeocodePlugin, see “Geographic Data Integration” in the Integration Guide.

Working with the Geocode Distributed Work Queue

ClaimCenter provides the Geocode distributed work queue, which regularly searches for new and changed addresses to geocode. Also, ClaimCenter provides the Geocode batch process, which you can run at any time from the command line.
The Geocode work queue comprises a writer daemon and a set of worker daemons. In a clustered environment, the work queue infrastructure improves overall throughput because it can divide the work into parallel geocoding processes. The writer daemon runs on the batch server, but worker daemons run on any of the servers in the cluster.
Understanding the Geocode Work Queue Process
The Geocode writer daemon searches the database for Address instances with the BatchGeocode property set to true and the GeocodeStatus property set to none. The writer adds addresses that qualify to a queue for geocoding. Worker daemons submit addresses from the work queue to the GeocodePlugin. After the GeocodePlugin adds geocode coordinates to an address, the worker daemon updates the address in the database.
An address sometimes changes during the interval between the writer adding a copy to the work queue and a worker updating the address in the database. For example, a user modifies a new address with additional street number or postal code information. If a worker daemon encounters an update conflict after geocoding an address, it submits the modified address to the GeocodePlugin and attempts to save the address again.
Performance Considerations with the Geocode Work Queue
At the time you first start ClaimCenter, your database might have many addresses to geocode, especially if you imported many new addresses into your production database. If your production database has many new addresses, the GeocodePlugin might take a long time to process these new addresses. Assure that you configure the Geocode work queue with a sufficient number of worker instances before you start your production servers.
Geocoding a large number of addresses is resource intensive. Assure that you configure the Geocode work queue to run during periods of minimal activity on the ClaimCenter servers.
The Geocode distributed work queue and the Geocode batch process can potentially update a large number of Address records in the database. After the Geocode work queue or the Geocode batch process completes, update database statistics by running the incrementaldbstats process.

Configuring Geocoding

Configuring geocoding in ClaimCenter involves enabling the GeocodePlugin, setting geocoding feature parameters, and scheduling the Geocode distributed work queue in ClaimCenter. If you use ContactManager, you must do the same.
Enabling the Geocode Plugin
By default, the GeocodePlugin plugin uses the Bing Maps implementation, but the plugin is disabled.
Before you can use the Bing Maps plugin implementation, your organization must have its own account, login, and application key with Bing Maps. For more information, go to http://www.bingmapsportal.com, where you can set up a Bing Maps account and obtain an application key. When you create an application key, the application name is arbitrary and no application URL is required.
To enable and use the Bing Maps plugin in ClaimCenter
1.
Start ClaimCenter Studio.
At a command prompt, navigate to ClaimCenter\bin and enter the following command:
gwcc studio
2.
Double-click to open configuration → config → Plugins → registry → GeocodePlugin.gws.
3.
Click the Enabled check box to enable the plugin.
4.
Assure the Class field specifies the Bing Maps implementation class:
gw.plugin.geocode.impl.BingMapsPlugin
5.
Under Parameters, specify:
 
applicationKey
The application key that you obtained from Bing Maps.
geocodeDirectionsCulture
The locale for geocoded addresses and routing instructions returned from Bing Maps. For example, use the locale code ja-JP for addresses and instructions for Japan. The plugin uses en-US if you do not specify a value. For a current list of codes that Bing Maps supports, see http://msdn.microsoft.com/en-us/library/cc981048.aspx.
imageryCulture
The language for map imagery. For example, use the language code ja for maps labeled in Japanese. The plugin uses en if you do not specify a value. For a current list of codes that Bing Maps supports, see http://msdn.microsoft.com/en-us/library/cc981048.aspx.
mapUrlHeight
Height of maps, in pixels. The plugin uses 500 if you do not specify a value.
mapUrlWidth
Width of maps, in pixels. The plugin uses 500 if you do not specify a value.
6.
Save your changes.
IMPORTANT If you also use ContactManager, repeat your changes in ContactManager Studio.
Setting Geocoding Feature Parameters
Configure geocoding features in the user interface with the following parameters in config.xml.
Parameter
Description
default
UseGeocodingInPrimaryApp
If true, ClaimCenter enables proximity search for users in the assignment user interface. ContactManager does not respond to this parameter.
false
UseGeocodingInAddressBook
Set to true if you have ClaimCenter integrated with ContactManager and ContactManager has geocoding enabled for vendors. This setting enables vendor search in the ClaimCenter and ContactManager user interfaces.
false
ProximitySearchOrdinalMaxDistance
The maximum distance to use if performing an ordinal (nearest n items) proximity search. This distance is in miles, unless you set UseMetricDistancesByDefault to true. This parameter has no effect on radius (within n miles or kilometers) proximity searches or walking-the-group-tree-based proximity assignment. Set this parameter identically in both ClaimCenter and ContactManager.
300
ProximityRadiusSearchDefaultMaxResultCount
The maximum number of results to return if performing a radius (within n miles or kilometers) proximity search. This parameter has no effect on ordinal (nearest n items) proximity searches. This parameter does not have to match the value of the corresponding parameter in the ContactManager config.xml file.
1000
UseMetricDistancesByDefault
If true, ClaimCenter uses kilometers and metric distances instead of miles and United States distances for driving distance or directions. Set this parameter identically in both ClaimCenter and ContactManager.
false
Scheduling and Configuring the Geocode Distributed Work Queue
You must edit the following files to schedule and configure the Geocode distributed work queue.
•
scheduler-config.xml
•
work-queue.xml
Access scheduler-config.xml in Guidewire Studio at configuration → config → scheduler → scheduler-config.xml. Access work-queue.xml in Guidewire Studio at configuration → config → workqueue → work-queue.xml.
Scheduling when the Geocode Batch Process Runs
You schedule the Geocode distributed work queue by modifying the following section in scheduler-config.xml file. 
<ProcessSchedule process="geocode">
  <CronSchedule hours="1" minutes="30"/>
</ProcessSchedule>
By default, the schedule for the Geocode work queue is disabled. To enable the schedule, uncomment the section. By default, the schedule runs the Geocode work queue at 1:30 AM daily. If you regularly add many new contacts, especially in ContactManager, tune the schedule to match your expected daily load of new addresses.
IMPORTANT Schedule the Geocode and the ABGeocode batch processes with sufficient processing windows between runs to assure sufficient time for runs to fully process the work items in the work queues. If you find duplicate work items in the work queues for the same address ID, extend the interval between runs.
Configuring the Number of Worker Instances for the Geocode Distributed Work Queue
You configure the Geocode distributed work queue by modifying the following section in the work-queue.xml file. 
<work-queue workQueueClass="com.guidewire.pl.domain.geodata.geocode.GeocodeWorkQueue"
 progressinterval="600000">
  <worker instances="1"/>
</work-queue>
The default work queue configuration specifies one Geocode worker instance. A worker instance passes addresses from the work queue to the GeocodePlugin and then updates the database. In a clustered environment, consider increasing the number of worker instances to improve throughput.
IMPORTANT The Geocode distributed work queue and the Geocode batch process can potentially update a large number of Address records in the database. After the Geocode work queue or the Geocode batch process completes, update database statistics by running the incrementaldbstats process.

Geocode Status

The GeocodeStatus typelist defines the set of status codes returned from the plugin. This typelist is final, so you cannot edit it. Access the GeocodeStatus typelist from Guidewire Studio at configuration → config → Metadata → Typelist → GeocodeStatus.tti.
For more details, “Geocoding Status Codes” in the Integration Guide.
