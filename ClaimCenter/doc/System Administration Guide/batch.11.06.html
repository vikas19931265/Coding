ClaimCenter System Administration Guide : Batch Processes and Work Queues : Configuring Distributable Work Queues

Configuring Distributable Work Queues

To configure a distributable work queue, you configure a schedule for the writer batch process, work queue attributes and worker attributes.
Configure the writer batch process schedule in the scheduler-config.xml file, just as you would any other batch process. Access this file in Guidewire Studio at configuration → config → scheduler → scheduler-config.xml. See “Scheduling Batch Processes and Work Queues”.
Configure work queue and worker attributes in the work-queue.xml file. Access this file in Guidewire Studio at configuration → config → workqueue → work-queue.xml.
After you edit either scheduler-config.xml or work-queue.xml, you must rebuild and redeploy ClaimCenter.
Each distributable work queue has its own configuration structure in work-queue.xml. 
<work-queue workQueueClass="string" progressinterval="decimal">
  <worker instances="integer" throttleinterval="decimal" orphansfirst="Boolean" env="string"
   server="string"/>
  ...
  <worker instances="integer" throttleinterval="decimal" orphansfirst="Boolean" env="string"
   server="string"/>
</work-queue>
The <work-queue> element has the following attributes:
 
Attribute
Description
workQueueClass
Required. The workQueueClass must be one of the Guidewire-provided work queue classes listed in work-queue.xml. These work queue classes implement the BatchProcess interface. You cannot configure custom batch processes as distributed work queues. You also cannot configure Guidewire-provided batch processes as distributed work queues if they are not already defined as distributable work queues in work-queue.xml.
progressinterval
Required. The progressinterval value is the amount of time, in milliseconds, that ClaimCenter allots for a worker to process batchsize work items. If the time a worker has held a batch of items exceeds the progressinterval, then ClaimCenter considers the work items orphans. ClaimCenter reassigns orphan work items to a new worker instance. The progressinterval must be greater than the time to process the slowest work item, or that work item will never be completed. Also, Guidewire recommends that you set progressinterval greater than the processing time of the whole batch. If a batch takes more time than progressinterval, ClaimCenter checks the remaining work items back in to the database. If there are many batches which take longer than progressinterval, the repeated checking out and checking in of work items can negatively impact performance.
retryInterval
Optional. How long in milliseconds to wait before retrying a failed item. The default value is 0, meaning ClaimCenter will retry processing the item immediately.
retryLimit
Optional. How many times ClaimCenter retries a failed work item for this work queue.
If you do not specify a retryLimit value for a work queue, ClaimCenter uses the value of the WorkItemRetryLimit configuration parameter in config.xml as the default value.
logRetryableCDCEsAtDebugLevel
Optional. If logRetryableCDCEsAtDebugLevel is set to true for a work queue, ClaimCenter logs any retryable Concurrent Data Change Exception (CDCE) at the DEBUG level. The log message includes a prepended string indicating that the error is considered to be non-fatal. Any CDCE that pushes the retry count over the retryLimit or workItemRetryLimit (if retryLimit is not set) is logged at the ERROR level.
For each work queue, you can declare as many worker instances as you want, specifying on which server and environment each one runs. If you do not specify a worker for a queue, or you do not specify the env and server attributes, ClaimCenter starts a single instance on the batch server. See “Specifying Environment Properties in the <registry> Element” for information about the env and server attributes. The worker subelements take the following attributes, all of which are optional:
 
Attribute
Description
instances
The number of workers to create. By default, ClaimCenter always creates at least one worker.
maxpollinterval
How often workers wake up automatically, even if the workers receive no notification. This default might not be appropriate for large numbers of workers. You might need to increase maxpollinterval to prevent excessive numbers of queries for work items. The default maxpollinterval is 60,000 milliseconds.
throttleinterval
The delay between processing workitems in milliseconds. The value controls how long the process sleeps. A value of 0 (zero) means worker processes work items as rapidly as possible. To reduce the CPU load, set the throttleinterval to a positive value.
batchsize
How many work items the worker attempts to check out while searching for more work items. Larger batch sizes are more efficient, but might not result in good load distribution. The default batchsize is 10.
env
The environment in which this particular worker is active.
server
The server on which this particular worker is active.
You can dynamically configure work queues by using web services. See “Manipulating Work Queues Using Web Services” in the Integration Guide.
