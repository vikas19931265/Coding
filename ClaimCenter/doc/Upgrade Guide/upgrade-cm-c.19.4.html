Manually Upgrading the ContactManager Configuration

Because ContactManager has changed a number of the files used to integrate with the Guidewire applications, it is likely that you will need to manually update configuration files. In particular, you will need to make manual updates:
•
If you have made changes to the ABContact data model.
•
If you have changed any of the configuration files that are listed in the table in “Configuration File Changes”.
In general, the steps for upgrading ContactManager are:
1.
Run the configuration upgrade tool and perform an automatic upgrade of the ContactManager configuration. See “Upgrading the ClaimCenter 6.0.x Configuration”.
2.
If you are using geocoding, before running the database upgrade, see Geocoding Updates.
3.
Run the database upgrade tool to upgrade the ContactManager database. See“Upgrading the ClaimCenter 6.0.x Database”.
4.
Manually configure files in ContactManager—the subject of this topic.
5.
Upgrade ClaimCenter as described at “Upgrading ClaimCenter from 6.0.x for ContactManager”.

Mapping Your Contact Extensions

If you have extended an ABContact entity or subentity with a new field or you have added a new ABContact subentity, you must update the ContactManager class gw.contactmapper.ab800.ContactMapper. The files you use to map these extensions to ClaimCenter have changed, and if you want to map your extensions, you must update these new files. For an overview of these mapping files, see “Configuration File Changes”.
The ContactMapper class maps entities to XML to send to core applications, and it maps them from the XML sent from core applications. You must map all fields that you want to receive from core applications or send to them.

Additional Search Files

This topic applies if you have extended ABContactSearchCriteria.etx with your own entries. For example, you extended an ABContact entity or subentity and you wanted to support the new fields in searches initiated by the Guiders core applications. The following new Gosu files support searches by Guidewire core applications:
•
gw.webservice.ab.ab800.abcontactapi.ABContactAPISearchCriteria – Provides search criteria as a web service to Guidewire applications. A Guidewire application uses this web service to determine which criteria to use in a contact Search.
•
gw.webservice.ab.ab700.abcontactapi.ABContactAPISearchResult – Defines search results sent to Guidewire applications as a web service. A Guidewire application uses this web service to interpret Search results that ContactManager returns after a search request from the application.
Note: If you added your own entries to search-config.xml in ContactCenter 6.0.x, accept the file in the configuration tool to copy it to the ContactManager 8.0.x configuration. There are no additional files related to this XML configuration file.
See also
•
“ContactManager Support for Contact Searches” on page 2 in the Contact Management Guide

Minimum Search Criteria

This topic applies if you previously specified that a Contact entity field was required in a search in the AdditionalPrimaryABContactSearchCriteria element in config.xml. This configuration parameter is no longer valid. Instead, you must make new entries in ValidateABContactSearchCriteriaPluginImpl.
See also
•
“Local ContactManager Search Criteria” in the Contact Management Guide

Definitive and Potential Matching Files

If you added entries to the file definitive-match-config.xml, you need to make new entries in at least one of the following files:
•
ValidateABContactCreationPluginImpl – A plugin implementation that enables you to provide your own logic for creating new users. In the base implementation, all contacts are required to have a contact tag. The base plugin implementation provides a set of more relaxed minimum creation requirements.
•
FindDuplicatesPlugin – Attempts to find exact and potential matches for the subtype of ABContact that is passed to it. If no Duplicate Finder exists for the subtype, the plugin implementation traverses the ABContact tree to find a parent type it can use in the search. The plugin implementation uses a set of Duplicate Finder classes, which define potential and exact match logic for a set of ABContact subtypes. For more information, see “IFindDuplicatesPlugin Plugin Interface” in the Contact Management Guide.
If you added entries to the file potential-match-config.xml, you need to make new entries in FindDuplicatesPlugin. As described previously, this class attempts to find exact and potential matches for the subtype of ABContact that is passed to it.
See also
•
“ValidateABContactCreationPlugin Plugin Interface” in the Contact Management Guide
•
“Duplicate Finder Classes” in the Contact Management Guide

Geocoding Updates

ContactManager provides a new implementation of the GeocodePlugin that connects with the Microsoft Bing Maps Geocode Service. In the default installation of ContactManager, the GeocodePlugin is disabled but is configured to use the BingMapsPlugin implementation. You must obtain a Bing Maps application key from Microsoft to use as credentials for this implementation.
In addition, the Address entity has a new BatchGeocode property that is used in combination with the GeocodeStatus property to determine if an address needs to be geocoded.
For a description of geocoding with Bing Maps, see “Geocoding and Proximity Searches for Vendor Contacts” in the Contact Management Guide. That topic describes the Bing Maps plugin and the BatchGeocode field.
To upgrade to support these geocoding changes:
•
If you already use geocoding, run the geocode process before upgrading to geocode all addresses that are not yet geocoded.
•
If you used geocoding in the past and want to use geocoding after you upgrade, write logic to set the BatchGeocode field. You can use the IDatabaseUpgrade plugin described at “Running Custom Version Checks and Triggers”.
•
Before deploying to a production server, upgrade your Microsoft MapPoint account to a Bing Maps account and edit the GeocodePlugin registry to use the BingMapsPlugin implementation.
