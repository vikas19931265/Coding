Manually Upgrading the ContactManager Configuration

Because ContactManager has changed a number of the files used to integrate with the Guidewire applications, it is likely that you will need to manually update configuration files. In particular, you will need to make manual updates:
•
If you have made changes to the ABContact data model.
•
If you have changed any of the files that are listed in “Manually Configuring Changed Files”.
In general, the steps for upgrading ContactManager are:
1.
Run the configuration upgrade tool and perform an automatic upgrade of the ContactManager configuration. See “Upgrading the ClaimCenter 8.0.x Configuration”.
2.
Run the database upgrade tool to upgrade the ContactManager database. See “Upgrading the ClaimCenter 8.0.x Database”.
3.
Manually configure files in ContactManager—the subject of this topic.
4.
Upgrade ClaimCenter as described at “Upgrading ClaimCenter from 8.0.x for ContactManager”.

•

Changes that Support Externally Specified Unique IDs for New Contacts
The ContactManager class gw.contactmapper.ab800.ContactMapper has new fields that support handling external unique IDs sent from core applications. If you have customized ContactMapper in your configuration, you must merge your changes manually into this class. See “Mapping Fields of a ContactManager Contact” in the Contact Management Guide.

BillingCenter Web Services Version Change

BillingCenter has changed the version of all its web services to bc801. Therefore, the ContactManager web services collection bc800.wsc, which is in wsi.remote.gw.webservice.bc, now uses the following path to access the WSDL file for BillingCenter:
${bc}/ws/gw/webservice/bc/bc801/contact/ContactAPI?wsdl
See:
•
“Step 1: Integrate ContactManager with BillingCenter” in the Contact Management Guide
•
“Configuring ContactManager-to-BillingCenter Authentication” in the Contact Management Guide

Driver’s License State Field Now Uses Jurisdiction.ttx

When you are editing a contact who is a Person subtype in ClaimCenter or an ABPerson subtype in ContactManager, there is a Driver's License section on the edit screen. Both ClaimCenter and ContactManager have a version of this edit screen. The State field on this screen now uses the Jurisdiction.ttx file to populate the drop-down list for the field. If you want to change the contents of the drop-down list, edit the Jurisdiction.ttx file in Studio and add or remove values from the outgoing category driving_lic.
If you have extended either screen, you must merge your changes and use Jurisdiction.ttx.
For example, the ClaimCenter PCF file ContactBasicsDV.Person.pcf uses DriversLicenseInfoInputSet.pcf. The State field on this InputSet is a TypeKeyInput widget with value Person.LicenseState. This field’s filter property has the following code to filter values from Jurisdiction.ttx:
VALUE.hasCategory(JurisdictionType.TC_DRIVING_LIC)
The ContactManager example is similar. The PCF file ContactBasicsDV.ABPerson.pcf has an unnamed InputSet that has two driver’s license fields. The State field is a TypeKeyInput widget with value (contact as ABPerson).LicenseState. This field’s filter property has the following code to filter values from Jurisdiction.ttx:
VALUE.hasCategory(JurisdictionType.TC_DRIVING_LIC)
